# 1.2.0 Plan ‚Äî surfaced freshness + optional semantic ranking

## Origin

A live head-to-head comparison run in this repo on 2026-07-10 (same task string,
same repo, same moment) found a concrete, reproducible gap on the one job both
neo-localmcp and a heavier competitor (GitNexus) actually share: turning a task
string into ranked context.

- GitNexus's `query` tool returned symbols from `neo_localmcp/tools.py`,
  `server.py`, `repo_memory.py`, `cli.py`, `benchmark.py` ‚Äî confidently, with
  line numbers, no inline warning. All five files were confirmed (`ls`) to no
  longer exist; they were renamed in the 2026-07-07/07-09 reorg. GitNexus's own
  `list_repos` separately reported this index as 71 commits behind HEAD, but
  that staleness signal never reached the `query` response itself.
- neo-localmcp's `prepare_context`, run against the identical task string,
  returned the current correct paths (`neo_localmcp/mcp/system.py`,
  `neo_localmcp/retrieval/repo_memory.py`), matching live HEAD.
- Separately, GitNexus's `query` is hybrid BM25 + semantic vector search with
  RRF fusion; `prepare_context` today is FTS keyword + regex-symbol matching
  only, with no semantic layer at all. On the shared job, that is the one
  legitimate capability gap worth closing.

Two threads fall out of this, and both are scoped tightly for this release
rather than expanded into a rewrite:

1. **Freshness is neo-localmcp's actual structural edge over a graph-index
   competitor, but it isn't surfaced where it matters.** `lookup()` already
   reports `last_indexed_at` (closed in #64), but `prepare_context` ‚Äî the tool
   actually used before an edit ‚Äî doesn't tell the caller how current its
   answer is relative to HEAD. Today's test is direct evidence of what happens
   when a *different* tool skips this.
2. **The keyword-only ranking gap is real** and should be closed as an
   additive, optional layer ‚Äî never a replacement for the deterministic FTS
   path, per this repo's standing "deterministic retrieval must never depend
   on Ollama" invariant.

**Open item to resolve before or alongside this plan, not silently skipped:**
1.1.1 is staged on `main` but not yet tagged/published (blocked on #65, not yet
designed). Default assumption for this plan: fold whatever of #65 is still
wanted into 1.2.0 (or explicitly re-defer it ‚Äî owner's call) and ship one
combined tag rather than stopping to cut a separate 1.1.1 release first. State
which you want before 12d starts.

## Phase breakdown

Phases run **in order, with an explicit pause between each** for a model/effort
switch. 12a stands alone. 12b and 12c share embedding plumbing ‚Äî don't start
12c before 12b lands. 12d is the release phase and depends on all prior phases
being green.

### 12a ‚Äî Surface index freshness inline in `prepare_context` ¬∑ **Fable, Medium**

Scope: `neo_localmcp/retrieval/repo_memory.py` (`index_repo`, `repo_metadata`
writes), `neo_localmcp/mcp/memory.py` (`context_prepare` output assembly).

- At index time, stamp a new `repo_metadata` key alongside the existing
  `last_indexed_at`/`indexer_version`/`eligible_files`: the git commit hash
  that was current when indexing completed (`indexed_commit`). Reuse whatever
  `repo_utils.py` already exposes for reading the current commit ‚Äî the
  `context_prepare` header already prints `git: branch=main commit=...`, so a
  git-info helper exists; don't add a second one.
- In `prepare_context`'s output, add one line comparing `indexed_commit`
  against the current HEAD commit: either an exact match ("index: fresh, at
  HEAD") or a commit count behind (`git rev-list --count
  <indexed_commit>..HEAD`), plus a pointer to `refresh_index` when non-zero.
  Deterministic, no Ollama involvement, no new dependency.
- A repo indexed before this change has no `indexed_commit` key yet ‚Äî degrade
  to the current behavior (no freshness line) rather than erroring; it backfills
  on the next index/refresh.
- Tests: index at commit A, make N commits, call `prepare_context`, assert the
  freshness line reports N commits behind; assert "fresh" wording at 0; assert
  no crash and no freshness line for a repo indexed before this change (missing
  `indexed_commit`).

### 12b ‚Äî Optional semantic ranking layer ¬∑ **Fable, High** ¬∑ Opus validates the never-blocks guarantee

Scope: `neo_localmcp/config.py` (new `ollama.embed_model`, default `None` ‚Äî
disabled unless explicitly configured), `neo_localmcp/ollama_client.py` (new
bounded `embed()` call), `neo_localmcp/retrieval/repo_memory.py` (new
`file_embeddings` table), `neo_localmcp/mcp/memory.py` (ranking blend).

- New config key mirrors `fast_model`/`summary_model`: `ollama.embed_model`,
  defaulting unset. Nothing changes for any existing install until a user
  explicitly sets it via `set-ollama` ‚Äî matches the "Ollama is optional, never
  required" invariant by construction, not by a runtime check bolted on after.
- New table: `file_embeddings(repo_id, path, model, content_hash, vector BLOB,
  created_at)`. Keyed on content hash the same way `section_summaries` already
  is, so a source-file change invalidates and regenerates its embedding,
  independent of any global cache-clear.
- Embeddings are computed lazily, only during `index_repo`/`refresh`, only when
  `embed_model` is configured **and** Ollama responds within the existing
  bounded health/connect timeouts. A cold/busy/missing Ollama at index time
  just skips embedding generation for that pass ‚Äî never blocks indexing, never
  errors the call.
- Ranking: FTS scoring in `context_prepare` remains the primary, always-on
  path, unchanged. When embeddings exist for a repo, compute cosine similarity
  between the task-string embedding and each **already-FTS-selected
  candidate's** stored vector, and blend it into that candidate's score as a
  re-ranking signal only ‚Äî this phase does not introduce a second
  candidate-generation path or expand the result set beyond what FTS already
  surfaced. Keeps the blast radius small and the fallback trivial: no
  embeddings anywhere in the repo ‚Üí ranking behaves exactly as it does today.
- **Opus's job before this phase closes**: adversarially verify the
  never-blocks guarantee ‚Äî kill Ollama mid-session, confirm `prepare_context`
  and `index_repo` still complete correctly and quickly on FTS alone; confirm
  `embed_model` unset produces byte-identical ranking output to today's
  baseline (regression, not just a new-behavior test).
- Acceptance evidence: run the existing `neo_localmcp/benchmarker`
  (`queries/default.jsonl`) once with `embed_model` unset and once configured,
  on this repo. Record the before/after retrieval-quality delta in
  `PROJECT_STATUS.md` ‚Äî this is real, reusable measurement infrastructure,
  not a new benchmark harness to build.

### 12c ‚Äî Semantic matching for retrieval-boost memory ¬∑ **Fable, Medium** (depends on 12b)

Scope: `neo_localmcp/retrieval/repo_memory.py` (`record_task_query`,
`get_boost_map`).

- Reuses 12b's `embed()`/embedding-storage plumbing ‚Äî no new infra. Each
  recorded `task_queries` row gets an embedding stored alongside it, only when
  `embed_model` is configured (same lazy, non-blocking rule as 12b).
- `get_boost_map`'s existing exact-`term_key` lookup stays the unchanged,
  zero-cost first path. Only as a fallback, when no exact match exists, scan
  other task-query embeddings in the same repo for cosine similarity above a
  fixed threshold, and apply a **down-weighted** fraction of that other task's
  accumulated boost ‚Äî never full credit, never overriding a real exact match.
- This is the concrete version of "recognize a paraphrased task as the same
  recurring workflow," the memory-gap-repair behavior discussed earlier: "fix
  login bug" and "debug auth failure" can now share partial boost instead of
  starting from zero every time the wording changes.
- Tests: record boost activity under task A ("fix login bug"); verify a
  semantically close but differently-worded task B ("debug auth failure")
  receives a nonzero but reduced boost; verify an unrelated task C gets none;
  verify `embed_model` unset leaves `get_boost_map` behaving exactly as today
  (regression test, not just new-path coverage).

## Deferred (explicitly, not forgotten)

- **Cross-repo/global pattern memory** ("this fix pattern recurred across
  multiple repos") ‚Äî a real idea, but it's a different data model
  (repo-scoped tables would need a cross-repo view) and has no driving
  evidence yet. Revisit only if a concrete need shows up.
- **Confidence decay for stale boosts** (a `retrieval_boost` entry that keeps
  getting `corrected` should shrink, not just cap) ‚Äî natural extension of 12c,
  deliberately left out here to keep this release's semantic-memory surface
  to one new capability (paraphrase matching), not two at once.
- **Tree-sitter structural symbol extraction** (replacing today's regex-based
  extractor) ‚Äî real known limitation, but unrelated to this release's theme
  and a large lift on its own; not pulled in just because it's adjacent.
- **`OLLAMA_MODELS` env-var inheritance bug in `start_service()`** ‚Äî known,
  tracked in `PROJECT_STATUS.md`, not part of this plan.
