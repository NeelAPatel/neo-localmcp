# neo-localmcp 1.0.10 Cross-Platform Setup v2 Implementation Plan

**Goal:** Replace the Windows-first installer family with one Python 3.12+ lifecycle that safely installs, reinstalls, uninstalls, clean-installs, migrates, reconnects clients, preserves memory by default, and verifies the resulting MCP endpoint on Windows and macOS (Linux deferred).

**Architecture:** `setup.py` (originally `setup_v2.py` during development, renamed in Task 16) is the sole lifecycle entrypoint and delegates to focused modules under `neo_localmcp/installer/`. Shared Python code owns policy; cross-platform libraries and small platform adapters handle executable paths, process details, permissions, and filesystem replacement. A candidate venv is built and verified before any healthy runtime is replaced.

**Tech Stack:** Python 3.12+, stdlib `argparse`/`venv`/`pathlib`/`json`/`subprocess`, `psutil` for cross-platform process ownership and trees, existing MCP/FastMCP client stack, pytest, isolated temporary homes, and real lifecycle smoke tests on macOS/Windows.

## Global Constraints

- `setup.py` is the sole supported install/reinstall/uninstall entrypoint.
- The installed executable remains `neo-localmcp`; no `neo` alias is created.
- Python 3.12 or newer is required; the installer never installs system Python.
- Installation source is the local checkout containing `setup.py`.
- `install` reuses durable data; `reinstall` replaces only the runtime; `uninstall` removes only the runtime by default.
- `install --clean` deletes the entire validated managed root and then installs fresh.
- `uninstall --delete-memory --yes` deletes the entire validated managed root and does not reinstall.
- Full-root deletion always requires interactive confirmation or the explicit `--yes` non-interactive flag.
- Durable data is never deleted by default.
- Neo unloads models it used but never kills a shared Ollama daemon.
- Process termination requires verified Neo ownership; never kill by process name alone.
- Avoid OS-specific files unless they materially improve reliability, stability, or functionality.
- When implementation depends on a library/tool API, consult Context7 first when available, then the product's primary official documentation; use third-party sources only when primary documentation is silent and label the inference.
- Every command and subcommand supports `-h` and `--help`.
- Implementation is test-first, each phase ends in a focused commit, and old installers remain available until real Windows parity is verified.

## Scope and sequencing

This plan was intentionally sequential because paths, state, runtime transactions, and ownership are safety boundaries consumed by every later task. Tasks 1-13 were built and verified on macOS. Task 14 used Windows x64 CI as its durable gate plus a temporary Windows VM for targeted live smoke testing. Task 15 added CI regression gates (Linux deferred by owner decision). Old scripts were retired in Task 16 after all gates passed.

## Documentation verification protocol

Before implementing an unfamiliar or version-sensitive API:

1. Resolve and query the official/high-reputation Context7 library entry.
2. Confirm safety-sensitive behavior against the upstream project's official docs
   or repository documentation when Context7 excerpts leave ambiguity.
3. Record the relevant documentation URL in task notes or a code comment only
   where it explains a non-obvious safety decision.
4. If primary documentation does not answer the question, state the inference and
   add a test that makes the assumed behavior explicit.

Documentation checks completed while writing this plan:

- psutil officially documents PID/create-time identity, `exe()`, `cmdline()`,
  descendant traversal, `terminate()`, `kill()`, `wait_procs()`, and handling of
  `NoSuchProcess`/`AccessDenied`: https://psutil.readthedocs.io/
- Ollama officially documents model unload through `POST /api/generate` with the
  model name and `keep_alive: 0`, without stopping the daemon:
  https://github.com/ollama/ollama/blob/main/docs/api.md
