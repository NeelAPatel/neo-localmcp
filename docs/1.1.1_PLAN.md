# 1.1.1 Plan — repeatable token-reduction benchmarking

## Origin

`estimated_tokens_returned` reports what a single response cost, but the
8-10x reduction figures quoted in `PROJECT_STATUS.md` came from one-off
manual comparisons during real audit sessions, not a repeatable tool. The
README's own north-star framing has listed "automated token-reduction
measurement" as a planned-but-not-built capability since day one (issue #9).

Design was worked out across several sessions (2026-07-06) rather than
guessed at up front; `docs/mcp-perf-log-issue-10.md`'s "Benchmarking
recommendations" section was written specifically to feed this and is the
other reference document alongside this plan. This document is
self-contained — every decision below reflects something actually settled
during that discussion, not a default I picked unilaterally.

(Originally drafted as `1.2.0_PLAN.md`; renamed 2026-07-06 once it became
clear this whole batch — audit-backlog upkeep plus this benchmark work —
ships as the 1.1.1 patch release, not a separate minor version. 1.2.0 has
no plan of its own yet.)

**Not yet designed, tracked separately in #65:** the live agent-vs-agent
comparison (spin up a real agent twice, with vs. without MCP tools, measure
real API-reported token usage) has its own open design questions (stopping
condition, cost ceiling, run-to-run variance) not resolved by this plan.
It is in scope for the 1.1.1 release, layering on top of this plan's report
format, but is a distinct piece of design/implementation work — not a
"1a-1f-are-done-therefore-this-is-done" checkbox.

## Decisions locked in during design (recap)

- **CLI-only**, `neo-localmcp benchmark <group...> [flags]` — administration
  is CLI-only, never an MCP tool, matching the existing convention for
  `index`/`refresh`/`reset-repo`/etc.
- **Named, extensible groups in a registry** — `sys`, `mem`, `ollama` today;
  `full` is computed as the union of every *registered* group, not a
  hardcoded list, so a future group joins `full` automatically just by being
  registered.
- **`full` costs "extremely minimal" tokens, not zero.** No group makes a
  live model call on its own, but reading potentially-large tool output back
  into an orchestrating agent's context is a real (small) cost worth naming
  honestly rather than calling it "free."
- **Never destructive.** Benchmark always targets the current repo (an
  explicit `--repo-root` is only required when there isn't one). Its only
  precondition is `refresh` (additive — updates stale/missing files, deletes
  nothing) — never `reset-repo`. Existing accumulated memory, however much
  or little exists, is untouched.
- **Never a silent side effect on real memory.** Benchmark-invoked queries
  must not be recorded into the real repo's `task_queries`/
  `retrieval_boost` history, so running the benchmark repeatedly can't
  quietly reshape future real retrieval just from being benchmarked.
- **Synthetic queries are mechanical, not LLM-generated**, in this plan —
  templated from real indexed symbols, zero token cost. This must be
  labeled explicitly in the report (which method produced each query),
  per the transparency requirement below. LLM-assisted generation was
  discussed but deferred (see Deferred section).
- **Determinism is a gate, not a metric.** Every `mem` query runs
  `test-determinism --runs 5` before being scored; a non-deterministic query
  is flagged and excluded from accuracy scoring, never silently averaged in.
- **`ollama` group tests reachability + configured-model presence +
  reliability** (repeated calls, success rate) — never requires the model to
  already be warm. It exercises the *existing* auto-load-if-cold behavior
  (`chat()` → `ensure()`, verified already in place in `ollama_client.py`)
  rather than reimplementing anything.
- **One persistent report per run.** Timestamped, never overwritten, never
  auto-deleted by the tool — only the user prunes old reports. Path
  convention: `neo-localmcp_benchmarks/<timestamp>/benchmark_<groups-and-flags>_<timestamp>.{json,md,csv}`,
  written under the current working directory (not inside whatever repo is
  being benchmarked, since that repo could belong to someone else).
- **Transparency is a hard requirement, not a nice-to-have.** The report
  must always be explicit about: which repo/commit was benchmarked, which
  groups ran, how each query was generated (mechanical vs. hand-curated),
  which queries were excluded for non-determinism, and that token figures
  are char÷4 estimates, not real telemetry, unless/until #65 exists.
- **Open question, explicitly not solved by this plan:** there's no agreed
  way to establish ground-truth "gold files" for natural-language queries.
  In v1, natural-language queries may have `gold_file: null`, in which case
  they're scored on speed/tokens only — no accuracy claim until this is
  resolved.

## Phase breakdown

All phases are similar risk/skill level; no model/effort switch needed
between them.

### 1a — CLI scaffold + group registry

Scope: new `neo_localmcp/benchmark.py`, `cli.py` wiring.

- `neo-localmcp benchmark <group...> [--repo-root] [--out <dir>]` subcommand.
- A small registry (`dict[str, list[Check]]`) mapping group name → ordered
  checks; `full` derived as the union of every registered group's checks at
  call time, never a separately-maintained list.
- Each check is a uniform record (name, what it runs, how to score the
  result) so `sys`/`mem`/`ollama` all plug into the same execution/reporting
  path instead of three separate ad hoc runners.
- Report-writing plumbing built and working end to end at this phase, even
  if the report is nearly empty — get the path convention, JSON shape, and
  "never overwrite" behavior right before there's much data flowing through
  it.

### 1b — `sys-only` group

Scope: benchmark module only; no new product code (reuses existing
`doctor`/`status`/`where`/`model status` etc. as-is).

- Each check runs one existing CLI command against the target repo, records
  wall time and pass/fail (did it error, not "was the answer good").
- No LLM, no Ollama — always safe and free to run.

### 1c — `mem-only` group: precondition + synthetic queries

Scope: benchmark module; **one small, explicit product change** —
`repo_memory.record_task_query` (and whatever calls it, e.g.
`context_prepare`) needs a way to be told "don't record this," defaulted to
recording everywhere except the benchmark's own calls. Behavior-preserving
everywhere else.

- Precondition: `refresh` (never `reset-repo`) against the target repo.
- Synthetic query generation: pick N real indexed symbols from the target
  repo's own symbol table, template one query per symbol (e.g.
  `"debug {symbol}: {symbol}"`), gold file = that symbol's defining file.
  Zero LLM cost — and the report must say `"generation_method": "mechanical"`
  per query, not just imply it.
- Determinism gate as described above.
- Scoring: rank-of-gold-file (MRR/precision-style), wall time,
  `estimated_tokens_returned`.

### 1d — `mem-only` group: natural-language queries

Scope: benchmark module + a checked-in default query file
(`neo_localmcp/benchmark_queries/default.jsonl`) for self-benchmarking this
repo; `--queries <path>` overrides it for a different target repo.

- Each row: `{"task": "...", "gold_file": "..." | null, "mode": "natural"}`.
  `gold_file: null` is allowed and scored on speed/tokens only, per the open
  question above.
- Default file ships with a handful of real queries about this repo,
  reusing known-good cases where sensible (e.g. the #22/#23/#24 repro
  queries already have well-understood expected answers).

### 1e — `ollama-only` group

Scope: benchmark module only; no new `ollama_client.py` logic (reuses
`status`/`ensure`/`chat` exactly as they exist today).

- Reachability check, configured-model-presence check, then N repeated
  lightweight calls through the existing `chat()`/`ensure()` path to measure
  success rate and response-time variance — scored as "did it respond
  usably," never "did it match last time."
- Deliberately never pre-warms the model itself; exercising the existing
  auto-load-if-cold behavior is the point of this group.

### 1f — Report finalization

Scope: benchmark module.

- Render the JSON records into the Markdown table and CSV views from the
  same source data.
- Summary block: medians per group/mode, and — only when `mem` ran — a
  token-reduction ratio checked directly against the README's stated
  acceptance targets (≥50% fewer discovery/read tokens, ≥30% fewer total
  tokens), explicitly labeled as estimated (char÷4), not real telemetry.

## Deferred (explicitly out of scope here)

- **`--live-agent-comparison`** — tracked entirely in #65. In scope for the
  1.1.1 release (see Origin above), but not designed or scheduled as a phase
  in this document yet; needs its own phase breakdown once its open
  questions are resolved.
- **Natural-language query ground-truth generation** — open question;
  `gold_file: null` is the escape hatch until it's solved.
- **LLM-assisted synthetic query generation** — v1 ships mechanical-only. If
  wanted later, it must ship with the token-cost confirmation prompt
  discussed during design, as its own follow-up.
- **CI integration** — this is an on-demand developer tool in this plan, not
  a CI gate.
