# 1.2.1 Plan — general model recommendations in the wizard

## Origin

Split out of issue #97 during 1.2.0's release dogfooding. While testing the
new `embed_model` wizard step (`docs/1.2.0_PLAN.md` 12d), the fast/summary/
embed model-picker pages turned out to have a real bug (the default
selection could silently fall back to an embedding-only model — fixed in
1.2.0) and a real gap: no guidance on *which* installed model is actually a
good choice for each role.

The fix was scoped, mocked through several iterations (styled HTML, then
plain ASCII once it was clear the real wizard has no TUI chrome to preview
honestly), and locked in with the owner: each model-picker page shows a
**RECOMMENDED** section (colorized, marking installed/not-installed) above
an always-visible, never-collapsed **ALL INSTALLED MODELS** list, and the
confirm page warns if a selected model isn't pulled yet.

That design has two possible engines behind it:

1. **General** — a small curated "these are generally good choices for this
   role" table, no machine inspection at all.
2. **Hardware-aware** — the same table, additionally filtered/ranked against
   detected GPU VRAM / system RAM (NVIDIA via `nvidia-smi`, Apple Silicon via
   `sysctl`, RAM-only fallback for AMD/Intel/unknown).

(2) is real but meaningfully harder — cross-vendor VRAM detection has no
reliable stdlib-only answer for AMD/Intel, and it's a second module/feature
on top of (1). Splitting them keeps this release single-themed and ships the
locked-in UI sooner: **this plan is (1) only.** (2) is tracked standalone in
issue #97 and can layer on top of this work later without changing the UI
again — only what feeds the "recommended" list.

## Phase breakdown

Phases run **in order**, with a pause between each for a model/effort switch.
12.1c depends on 12.1a+12.1b's data both existing. 12.1d is the release phase.

### 12.1a — Curated per-role recommendation table · **Sonnet, Medium**

Scope: new `neo_localmcp/installer/model_recommendations.py`.

- `recommend_models(role, installed_models) -> list[RecommendedModel]`
  (`role` is `"fast"` / `"summary"` / `"embed"`), returning up to 2 candidates
  from a small hardcoded `_ROLE_CANDIDATES` table (well-known, generally
  good default models per role — e.g. a small/fast chat model for `fast`, a
  larger code-capable model for `summary`, a dedicated embedding model for
  `embed`). Each `RecommendedModel` carries `name`, and `installed: bool`.
- Tag matching reuses `ollama_client._resolve_installed_model` (already
  handles the `name` vs `name:latest` vs unambiguous-tag cases) rather than
  a second matching implementation.
- No hardware inspection anywhere in this module — that distinction is the
  entire point of this plan vs. issue #97.
- Tests: a candidate that's installed reports `installed=True` and resolves
  to its exact installed tag; a candidate that isn't installed reports
  `installed=False` and keeps the bare candidate name (what the user would
  `ollama pull`); each role returns at most 2 recommendations; an unknown
  role returns an empty list rather than raising.

### 12.1b — Model-picker pages render the locked-in layout · **Sonnet, Medium**

Scope: `neo_localmcp/installer/wizard/console.py` (`_phase_ollama_fast`,
`_phase_ollama_summary`, `_phase_ollama_embed`).

- Each page's list becomes: hint block (unchanged) → **RECOMMENDED FOR
  YOUR SETUP** section, one line per candidate from 12.1a, colored via the
  existing `_ansi` palette (`green` + `[x] available` when installed,
  `red_bold` + `[ ] not pulled` when not) → divider → **ALL INSTALLED
  MODELS**, the complete existing list, unchanged, never collapsed behind a
  "show all" option. Reuses `_format_model_table`'s existing kind-label
  (chat/embed) column and current/default-row marking from 1.2.0 — this is
  an additive section, not a replacement of that table.
- The numbered choices span both sections in one continuous sequence (the
  user still just enters a number), matching the approved mockup.
- Tests: scripted console-phase tests (matching the existing pattern from
  1.2.0's wizard fixes) asserting the recommended section appears with
  correct installed/not-installed markers, and the full installed list is
  still present and unabridged below it.

### 12.1c — Confirm-page pull reminder · **Sonnet, Medium**

Scope: `console.py`'s `_confirm()`.

- If any of the selected fast/summary/embed models isn't installed, the
  review-and-confirm page shows a `red_bold` warning block (matching the
  existing full-wipe-warning visual weight) listing the exact
  `ollama pull <model>` command for each one, before the final "Apply as
  shown?" question. Absent entirely when every selection is already pulled.
- Never blocks proceeding — Ollama's own graceful degradation (deterministic
  ranking continues, that model just reports "model missing" until pulled)
  already handles the runtime case; this is purely a heads-up so a user
  doesn't discover it by surprise later.
- Tests: all-selected-models-installed → no warning block; one or more
  missing → warning block lists exactly the missing ones with correct pull
  commands, nothing else.

### 12.1d — Release: version bump, green CI, merge, tag · **Sonnet drives, Opus final audit**

Scope: `neo_localmcp/__init__.py`, `pyproject.toml`,
`packages/claude-desktop/mcpb/manifest.json`, `README.md`, `PROJECT_STATUS.md`,
`PROJECT_NOTES.md`.

- Bump version to `1.2.1` in lockstep across the three files that must
  always match. Rebuild the `.mcpb` bundle, verify byte-identical to source.
- Full suite green (`python -m pytest -q`, `python -m compileall -q
  neo_localmcp setup.py`) on macOS and Windows CI.
- **Opus's final pass**: verify by hand against a real installed-model list
  (not just the unit-test fixtures) that the recommended section and the
  confirm-page warning both render correctly and that picking a
  recommended-but-not-installed model genuinely produces the expected
  warning at confirm time — the exact live-verification style used for
  1.2.0's release gate.
- Update `PROJECT_STATUS.md`/`PROJECT_NOTES.md` per the standing convention.
- Branch → PR → confirm CI green on the PR itself → merge commit (never
  squash/rebase) → `git tag v1.2.1` → push once explicitly confirmed.

## Deferred (explicitly, not forgotten)

- **Hardware-aware recommendations** (VRAM/RAM-based filtering on top of
  this plan's static table) — tracked standalone in issue #97, layers onto
  12.1a's data without changing the UI built here.
- **Brand consistency** — owner flagged this as a related but distinct
  concern to be scoped into this plan later; deliberately left unfilled here
  rather than guessed at.
