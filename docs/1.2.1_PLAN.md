## Origin

This release bundles four items surfaced during 1.2.0 dogfooding and its
follow-up sessions, run sequentially with testing between each, in severity
order:

1. **#100** — `neo-localmcp` was never put on `PATH` by the wizard/CLI
   install lifecycle, so the bare command is unusable after a normal install
   unless `PATH` already happens to include the managed venv's `bin/` some
   other way. **Highest severity: breaks the CLI universally, immediately
   post-install, on every platform, with no workaround short of manually
   diagnosing PATH resolution.**
2. **#103** — `setup_wizard.py`/`setup.py` only offer client selection
   (Claude Code / Claude Desktop / Codex) on Install; Reinstall and
   Uninstall never ask, and drift from each other. Worse, `_record_client_intent`
   silently discards an explicit client selection made *during Install
   itself* whenever the install isn't perfectly clean/fresh, falling back to
   a disk snapshot instead — with no separate menu to catch or notice this
   in the moment, unlike Reinstall/Uninstall's gap (which is at least
   coverable via the existing "Manage connected clients" menu once
   noticed). Uninstall's backend is also all-or-nothing regardless of any
   flag, so a partial client removal isn't currently possible at all via
   Uninstall directly.
3. **#68** — `ollama_client.py`'s `start_service()` spawns `ollama serve`
   without an explicit `env=`, so `OLLAMA_MODELS` (and other Ollama env
   vars) can fail to reach the spawned process under some launch ancestries
   (e.g. Claude Desktop's extension host), silently pointing Ollama at the
   wrong models directory. Only reachable when the configured Ollama
   `base_url` is local — remote (Tailscale, etc.) setups are unaffected,
   since `start_service()` refuses to touch a non-local `base_url` at all —
   and the symptom (`model_missing`) is at least diagnosable, even if the
   cause isn't obvious. Narrowest blast radius of the three fixes, hence
   lowest priority among them.
4. **#101** — general (hardware-blind) per-role model recommendations in the
   wizard's model-picker pages: a curated table, no machine inspection.
   Split out of #97 during 1.2.0's dogfooding; the hardware-aware half
   (VRAM/RAM-based filtering on top of this) stays tracked standalone in
   #97 and layers on later without touching the UI built here. A feature,
   not a bug fix — runs last.

**Priority order for this release: #100 -> #103 -> #68 -> #101**, each fully
tested (automated + manual, see below) before moving to the next.
**Deferred, no version assigned**: "brand consistency" (owner-flagged during
1.2.1 planning as related-but-distinct; deliberately left unscoped rather
than guessed at) and #97's hardware-aware layer.

## Branching / PR structure

Given this repo's CI is wired as `on: push: branches: [main]` +
`on: pull_request` (any base branch) — a bare push to a non-`main` feature
branch triggers **no** CI at all, but opening a PR against *any* base branch
does — the following structure gets real CI signal per issue while still
landing as a single reviewable change on `main`:

1. Cut `dev/v1.2.1_implementation` from `main`.
2. One branch + one PR per issue, each **merging into
   `dev/v1.2.1_implementation`**, not `main`:
   `fix/100-cli-path` -> `dev/v1.2.1_implementation`,
   `fix/103-client-selection-parity` -> `dev/v1.2.1_implementation`,
   `fix/68-ollama-env-inheritance` -> `dev/v1.2.1_implementation`,
   `feat/101-model-recommendations` -> `dev/v1.2.1_implementation`.
   Each gets the automated suite + its own manual verification pass (below)
   before merging into the integration branch.
3. Once all four are merged into `dev/v1.2.1_implementation` and green,
   open **one** PR from `dev/v1.2.1_implementation` -> `main`. That's the
   only PR that actually needs to satisfy `main`'s branch protection
   (`Validate PR title` + `ci-gate` required checks, `enforce_admins: true`)
   — the sub-branch merges into the integration branch are not gated by
   protection (only `main` has protection configured), so CI green on each
   sub-PR is a self-enforced discipline, the same "confirm CI is green,
   don't assume" rule this repo already applies to `main`, just applied one
   level earlier by hand.
4. No merge into `main` until all four issues are cleared, including any
   bugs discovered mid-implementation — the single `main`-bound PR is the
   release gate.

## Phase breakdown

Phases run **in order** per the priority above, with a pause between each
for manual verification (and a model/effort switch where noted).

### 12.1a — Fix #100: install lifecycle surfaces a PATH hint · **Sonnet, Medium**

Scope: `neo_localmcp/installer/operations.py` / `runtime.py` (wherever
install/reinstall completion is reported) + CLI (`installer/cli.py`) +
wizard (`installer/wizard/console.py`, `live_backend.py`) success screens.

- On successful install/reinstall (CLI and wizard), print the exact
  `export PATH=...` line for `~/.neo-localmcp/venv/bin` (POSIX) or the
  Windows `Scripts\` equivalent. Detect `$SHELL`/OS and offer to append it
  to the relevant rc file, gated behind explicit user confirmation — this
  repo's install lifecycle does not silently mutate files outside
  `~/.neo-localmcp` today, and that stays true here.
- Tests: install-completion output includes the correct PATH line for the
  current OS; the offer-to-append path only writes when confirmed; declining
  leaves rc files untouched.
- Manual: see Manual testing checkpoints above.

### 12.1b — Fix #103: standardize client selection across install/reinstall/uninstall · **Sonnet, Medium-High**

Scope: `neo_localmcp/installer/wizard/console.py` (`OP_REINSTALL`,
`OP_UNINSTALL` phase lists), `neo_localmcp/installer/cli.py` (`reinstall`,
`uninstall` subparsers), `neo_localmcp/installer/operations.py`
(`_record_client_intent`), `neo_localmcp/installer/clients.py`
(`remove_active_registrations`).

- Wizard: give `OP_REINSTALL` and `OP_UNINSTALL` their own `_phase_clients`
  step (pre-filled with currently-registered clients as the default), same
  as `OP_INSTALL`/`OP_MANAGE_CLIENTS` already do.
- CLI: add the same repeatable `--client {claude-code,codex,claude-desktop}`
  flag to `reinstall` and `uninstall` that `install`/`manage-clients`
  already have; unspecified = keep current set on reinstall, unspecified =
  remove all on uninstall (unchanged default, now overridable).
- Fix `_record_client_intent` (`operations.py:353`) so an explicit user
  selection is never silently overridden by a disk snapshot, regardless of
  `fresh`/install-state.
- Add a `remove_active_registrations` variant taking an explicit client
  subset instead of always removing everything recorded.
- Tests: reinstall/uninstall with a partial `--client` subset changes only
  the specified clients' live config; an explicit Install-time selection
  survives even over non-`ABSENT` leftover state; wizard Reinstall/Uninstall
  screens show the same three-way multi-select as Install.
- Manual: see Manual testing checkpoints above.

### 12.1c — Fix #68: explicit `OLLAMA_MODELS` (and related env) pass-through · **Sonnet, Medium**

Scope: `neo_localmcp/ollama_client.py`'s `start_service()`.

- Read `OLLAMA_MODELS` (and any other Ollama-relevant env vars worth
  covering) explicitly at neo-localmcp's own process level rather than
  relying on ambient inheritance; pass it through explicitly via `env=` on
  the `Popen([executable, "serve"], ...)` call.
- Build a repro harness that spawns neo-localmcp's own env-reading path
  under a deliberately sanitized-env subprocess, since this bug does not
  reproduce in a normal shell.
- Non-goals unchanged from the issue: no change to "never auto-download a
  missing model" behavior, no change to the remote-`base_url` refusal.
- Tests: sanitized-env harness confirms `OLLAMA_MODELS` still reaches the
  spawned `ollama serve` process; existing local-start/remote-refusal
  behavior unchanged (regression coverage).
- Manual: see Manual testing checkpoints above (Windows local-Ollama case).

### 12.1d — #101: Curated per-role recommendation table · **Sonnet, Medium**

Scope: new `neo_localmcp/installer/model_recommendations.py`.

- `recommend_models(role, installed_models) -> list[RecommendedModel]`
  (`role` is `"fast"` / `"summary"` / `"embed"`), returning up to 2
  candidates from a small hardcoded `_ROLE_CANDIDATES` table. Each
  `RecommendedModel` carries `name`, and `installed: bool`.
- Tag matching reuses `ollama_client._resolve_installed_model` rather than a
  second matching implementation.
- No hardware inspection anywhere in this module — that distinction is the
  entire point of #101 vs. issue #97.
- Tests: a candidate that's installed reports `installed=True` and resolves
  to its exact installed tag; a candidate that isn't installed reports
  `installed=False` and keeps the bare candidate name; each role returns at
  most 2 recommendations; an unknown role returns an empty list rather than
  raising.

### 12.1e — #101: Model-picker pages render the locked-in layout · **Sonnet, Medium**

Scope: `neo_localmcp/installer/wizard/console.py` (`_phase_ollama_fast`,
`_phase_ollama_summary`, `_phase_ollama_embed`).

- Each page's list becomes: hint block (unchanged) → **RECOMMENDED FOR
  YOUR SETUP** section, one line per candidate from 12.1d, colored via the
  existing `_ansi` palette (`green` + `[x] available` when installed,
  `red_bold` + `[ ] not pulled` when not) → divider → **ALL INSTALLED
  MODELS**, the complete existing list, unchanged, never collapsed. Reuses
  `_format_model_table`'s existing kind-label (chat/embed) column and
  current/default-row marking from 1.2.0 — additive, not a replacement.
- The numbered choices span both sections in one continuous sequence.
- Tests: scripted console-phase tests asserting the recommended section
  appears with correct installed/not-installed markers, and the full
  installed list is still present and unabridged below it.

### 12.1f — #101: Confirm-page pull reminder · **Sonnet, Medium**

Scope: `console.py`'s `_confirm()`.

- If any selected fast/summary/embed model isn't installed, the
  review-and-confirm page shows a `red_bold` warning block (matching the
  existing full-wipe-warning visual weight) listing the exact
  `ollama pull <model>` command for each, before the final "Apply as
  shown?" question. Absent entirely when every selection is already pulled.
- Never blocks proceeding.
- Tests: all-selected-models-installed → no warning block; one or more
  missing → warning block lists exactly the missing ones with correct pull
  commands, nothing else.
- Manual: see Manual testing checkpoints above.
