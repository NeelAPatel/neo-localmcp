# 1.2.2 Plan — onboarding distribution + enforcement

## Origin

**Renamed from `docs/1.2.1_PLAN.md` (2026-07-12).** `v1.2.0` shipped, and a
separate, single-themed general-model-recommendation wizard feature was
slotted into `1.2.1` instead (see `docs/1.2.1_PLAN.md`) — this plan's content
is unchanged, only the version number and phase prefixes (`12.1x` → `12.2x`)
shifted to make room. Same precedent as the `1.2.0_PLAN.md` → `1.1.1_PLAN.md`
rename earlier in this project's history (`PROJECT_NOTES.md`, 2026-07-06).

`v1.2.0` (surfaced freshness + optional semantic ranking, `docs/1.2.0_PLAN.md`)
is tagged and released. `v1.1.1` is tagged. `v1.0.10` was the only tag before
that.

Separately, while designing 1.2.0, a second, unrelated gap surfaced:
[MCP_AGENT_INTEGRATION.md](../MCP_AGENT_INTEGRATION.md) — built in 1.0.9's
phase 9a — is real, but structurally incomplete as an onboarding mechanism:

- **It's never distributed.** It lives only in this repo. Nothing copies it,
  or a version of it, into a repo that gets indexed.
- **Its paste-into-`CLAUDE.md` snippet is entirely manual and unenforced.**
  Today's flow is "here's a snippet, remember to paste it" — easy to skip,
  and skipping it silently means an agent never discovers `prepare_context`
  exists in that repo at all.
- **The installer/wizard never mentions it.** Post-install output nudges
  toward `neo-localmcp index` (1.0.9 phase 9d) but says nothing about wiring
  an agent up to actually use the index once it exists.

This is a distribution/enforcement gap, not a retrieval-quality gap — it
doesn't move `prepare_context`'s ranking or freshness at all, it moves whether
a user's agent ever gets pointed at the tool in the first place. Scoped as its
own release rather than folded into 1.2.0 to keep both plans single-themed.

## Phase breakdown

Phases run **in order, with an explicit pause between each** for a
model/effort switch. 12.2a must land before 12.2b/12.2c (they consume its
template). 12.2e is the release phase and depends on all prior phases.

### 12.2a — Single-source template for the agent-integration doc · **Fable, Medium**

Scope: new `templates/agent_integration/` (alongside the existing
`templates/claude-code/commands/neo-localmcp/` convention),
`MCP_AGENT_INTEGRATION.md`.

- Extract `MCP_AGENT_INTEGRATION.md`'s two existing sections (paste-able
  snippet, full tool reference) into one template source that can render two
  targets: this repo's own `MCP_AGENT_INTEGRATION.md` (dogfooding copy, kept
  for continuity) and a new per-repo `Usage_Instructions.md` (three-part:
  bootstrap block, usage loop, command reference — parts 2 and 3 are close to
  a direct reuse of the existing tool reference and paste-able snippet).
- Single source of truth is the point: today's file and tomorrow's
  per-repo copies must never hand-drift out of sync with each other.
- No behavior change yet — this phase is purely "stop hand-duplicating text,"
  verified by a test asserting the rendered dogfooding copy is byte-identical
  to today's `MCP_AGENT_INTEGRATION.md` content.

### 12.2b — `bootstrap-repo` CLI command · **Fable, High** · Opus validates idempotency

Scope: `runtime_cli.py` (new `bootstrap-repo` subcommand — CLI-only, per the
standing "administration is never an MCP tool" rule), `ai_client_config.py`
(reuse `_replace_marked_block`, the existing `# BEGIN neo-localmcp` /
`# END neo-localmcp` marker pattern already used for Codex's `config.toml` —
don't invent a second injection mechanism).

- `neo-localmcp bootstrap-repo --repo-root <path>`:
  - Detects `CLAUDE.md` or `AGENTS.md` in the target repo (prefers an
    existing one; creates `CLAUDE.md` only if neither exists).
  - Injects/updates the paste-able snippet via the marker-block pattern —
    idempotent by construction: running it twice produces byte-identical
    output, never a duplicate block.
  - Writes/refreshes `Usage_Instructions.md` at the target repo root from
    12.2a's shared template.
  - Entirely deterministic; touches no source code; requires no Ollama.
- **Opus's job**: run it twice in a row against the same repo and assert
  byte-identical results (true idempotency, not just "didn't crash"); run it
  against a repo whose `CLAUDE.md` already has unrelated hand-written content
  and assert nothing outside the marker block is touched.
- Tests: fresh repo with neither file → creates `CLAUDE.md` with the block;
  repo with an existing `CLAUDE.md` with other content → block inserted,
  everything else untouched; re-run → no diff.

### 12.2c — Deterministic self-shedding bootstrap section · **Fable, Medium**

Scope: `Usage_Instructions.md` template rendering logic (part of 12.2a/b's
code, not the agent).

- Rather than asking an agent to hand-edit `CLAUDE.md` and then judge which
  lines of `Usage_Instructions.md` to delete itself, `bootstrap-repo` decides
  this deterministically: it checks whether the target `CLAUDE.md`/`AGENTS.md`
  already contains the marker block. If not yet applied, the rendered
  `Usage_Instructions.md` includes Part 1 ("run `neo-localmcp bootstrap-repo`
  in this repo"). Once applied, regenerating the file omits Part 1 entirely —
  Parts 2 and 3 (usage loop, command reference) remain always-present.
- This keeps the "self-consuming first step" idea from the original proposal,
  but the agent's actual job is "run one command," not freeform text surgery
  on a doc it's simultaneously trying to follow.
- Tests: render before bootstrap applied → Part 1 present; render after →
  Part 1 absent, Parts 2/3 unchanged.

### 12.2d — Installer/wizard post-install summary surfaces the command · **Fable, Medium**

Scope: `installer/wizard/` (both `preview_backend.py`/`live_backend.py` per
the existing `WizardBackend` parity requirement), `setup.py`'s install
summary output.

- After a successful install (fresh or upgrade), print a called-out block —
  same emphasis style as 1.0.9 phase 9d's indexing nudge, not one bullet
  buried in a generic "Next:" list: "For each repo you want an agent to use
  this with, run `neo-localmcp bootstrap-repo --repo-root <path>` — wires up
  CLAUDE.md/AGENTS.md and drops a Usage_Instructions.md your agent reads to
  onboard itself."
- Both the scriptable CLI installer and the interactive wizard need this,
  per the existing CLI/wizard parity convention (`PROJECT_STATUS.md`,
  2026-07-06 entry) — one shared summary-rendering path, not two hand-written
  copies.

### 12.2e — Release: version bump, green CI, merge, tag · **Fable drives, Opus final audit**

Scope: `neo_localmcp/__init__.py`, `pyproject.toml`,
`packages/claude-desktop/mcpb/manifest.json`, `README.md`, `PROJECT_STATUS.md`,
`PROJECT_NOTES.md`.

- Bump version to `1.2.2` in lockstep across the three files that must always
  match. Rebuild the `.mcpb` bundle, verify byte-identical to source.
- Full suite green (`python -m pytest -q`, `python -m compileall -q
  neo_localmcp setup.py`) on macOS and Windows CI.
- **Opus's final pass**: adversarially confirm `bootstrap-repo` is genuinely
  safe to run against a real repo with pre-existing, unrelated `CLAUDE.md`
  content — not just the unit-test fixture case. Confirm the wizard/CLI
  summary text actually renders in both the fake and real backend.
- Update `PROJECT_STATUS.md`/`PROJECT_NOTES.md` per the standing convention.
- Branch → PR → confirm CI green on the PR itself → merge commit (never
  squash/rebase) → `git tag v1.2.2` → push once explicitly confirmed.

## Deferred (explicitly, not forgotten)

- **Other agent-rule surfaces** (Cursor `.cursorrules`, Windsurf rules, etc.)
  beyond `CLAUDE.md`/`AGENTS.md` — no evidence of demand yet, adds real
  per-surface maintenance; revisit if it comes up.
- **Auto-detecting which AI surface is in use** and tailoring the injected
  snippet's wording per-surface — today's snippet is surface-agnostic by
  design; only worth specializing if a concrete surface-specific problem
  shows up.
