# 1.0.9 Plan — onboarding polish + trustworthy memory

## Origin

1.0.8 shipped a clean single-venv-per-version upgrade cycle, but the *first-run*
experience is still rough: `setup.ps1` installs the CLI only, says nothing about
Ollama readiness, doesn't tell you what paths on your machine it's about to touch,
and its uninstall path is a single "remove everything CLI-side" flow with no way to
target Claude Desktop/Claude Code/Codex independently or choose what to keep. There's
also no standing instruction a downstream repo's `CLAUDE.md`/`AGENTS.md` can carry to
get an agent to actually discover and use neo-localmcp. Separately, the P5
retrieval-boost feature (deferred since 1.0.6) is live-wired into `context_prepare`/
`file_excerpts` but has never been calibrated or proven to survive an upgrade —
until that's checked, memory is a liability an upgrade could silently corrupt or
reset without anyone noticing, not an asset. Fixing the onboarding gaps first (1-6)
gives P5 (7) something stable to be tuned against.

## Phase breakdown

Phases run **in order, with an explicit pause between each** for a model/effort
switch. Do not start the next phase without the user resuming. 9a-9d all use the
same model/effort and can run back-to-back without a pause if preferred — only
9e and 9f/9g are actual switches.

### 9a — Copy-pasteable agent-integration snippet · **Sonnet, Medium**

Scope: new `docs/AGENT_INTEGRATION.md` (or similar), README pointer to it.

- A short block meant to be pasted into a *downstream* repo's `CLAUDE.md`/`AGENTS.md`
  — not neo-localmcp's own docs. It tells an agent working in that repo: before
  broad search, read neo-localmcp's own README/tool docs, decide concretely where
  in *this* project it's the right tool (repo-wide context prep vs. targeted lookup
  vs. summarization), and report that back to the user rather than silently deciding.
  Keep it to a few lines — this is a pointer, not a tutorial.
- Reuse language from the existing "Recommended agent instruction" block in
  `README.md` rather than inventing new phrasing from scratch.

### 9b — Ollama presence/model check at install · **Sonnet, Medium**

Scope: `install.ps1`.

- Detect `ollama.exe` via `Get-Command`. If absent, print a short note (Ollama is
  optional — deterministic retrieval works without it) and move on, no failure.
- If present and not already running, start `ollama serve` headless the same way
  `ollama_client.py`'s `start_service()` does, so first-run parity matches what the
  server itself would do.
- Run `ollama list`, diff against the config's `fast_model`/`summary_model`
  (defaults from `config.py`), and report per-model found/missing. Never
  auto-pull a missing model — matches the existing "a missing model is never
  downloaded automatically" rule from `ollama_client.py`.

### 9c — Versioned `.mcpb` packaging · **Sonnet, Medium**

Scope: `scripts/build-mcpb.ps1`, `scripts/build-mcpb.sh`, `install.ps1`, `setup.ps1`.

- Output filename becomes `neo-localmcp-v<version>.mcpb` (version read the same way
  `install.ps1`'s `Get-SourceVersion` already does), committed to
  `packages/claude-desktop/` each release instead of the current fixed
  `neo-localmcp.mcpb`. Keep shipping a copy to `~/.neo-localmcp/` on install so
  `setup.ps1`'s Desktop-extension instructions can point at a stable local path
  without the user hunting for a version-suffixed filename.
- Any place currently hardcoding `neo-localmcp.mcpb` (installer, wizard status
  output, README) needs to resolve the versioned name instead.

### 9d — Post-install indexing nudge · **Sonnet, Medium**

Scope: `install.ps1` end-of-run output, `setup.ps1`'s `Invoke-Install`.

- After a successful install (fresh or upgraded, not the same-version no-op path),
  print a strong, specific prompt: run `neo-localmcp index --repo-root <path>` in
  each repo you intend to use it on now, or ask your AI agent to do it (referencing
  9a's snippet). Currently this is buried as step 3 of a generic "Next:" list in
  `install.ps1` — make it its own called-out block, not one bullet among four.

### 9e — Installer surface selection with path preview · **Sonnet, High**

Scope: `setup.ps1` (`Invoke-Install`), `neo_localmcp/client_setup.py` (read-only
status calls already exist via `client_status()` — reuse, don't duplicate).

- Before installing, ask (multi-select, comma-separated numbers like the existing
  y/N prompts) which surfaces to set up: CLI (implicitly always, it's the
  prerequisite for the others), Claude Code, Codex CLI/Desktop, Claude Desktop.
- For each one selected, print the exact path(s) it will touch (pull from
  `client_status()`'s `paths` dict) *before* asking for final confirmation — e.g.
  Claude Code shows `~/.claude/commands/neo-localmcp/` + the `claude mcp add`
  scope, Codex shows `~/.codex/config.toml`, Claude Desktop shows it cannot be
  automated and states the manual `.mcpb` install path from 9c.
- This is additive to the existing CLI install/upgrade/repair logic in
  `Invoke-Install` — don't rewrite the venv logic itself, just gate the existing
  "register clients?" prompt at the end into per-surface selection instead of one
  all-or-nothing "Y/n".

### 9f — Uninstaller overhaul: multi-select surfaces + granular data keep/delete · **Opus, High**

Scope: `setup.ps1` (`Invoke-Uninstall`), new functions in
`neo_localmcp/client_setup.py` (`remove_claude_code()`, `remove_codex()` — neither
exists yet, only `setup_*` registration functions do), `uninstall.ps1`.

This is the highest-risk phase (destructive, multi-surface, partial-failure needs
to be handled per-surface rather than all-or-nothing) — why it's the one pulled to
Opus, same reasoning as 7b's graceful-shutdown phase in 1.0.7.

- Multi-select which surfaces to act on: Claude Code, Codex CLI/Desktop, Claude
  Desktop, CLI. For each selected:
  - **Claude Code**: run `claude mcp remove neo-localmcp --scope user` (mirrors the
    migration-removal pattern already in `client_setup.py:109`), remove the
    `~/.claude/commands/neo-localmcp/` directory. New `remove_claude_code()`.
  - **Codex CLI/Desktop**: strip the `# BEGIN neo-localmcp` / `# END neo-localmcp`
    marked block from `~/.codex/config.toml` (reuse `_replace_marked_block`'s
    delimiters, replace with empty instead of a new block). New `remove_codex()`.
  - **Claude Desktop**: cannot be automated (by design — `client_setup.py:154`,
    "Direct claude_desktop_config.json editing is intentionally no longer
    performed"). Print the existing `Show-DesktopExtensionNotice` guidance plus a
    reminder that menu option 3 (`Invoke-StopDesktopExtension`) fixes the common
    stuck-uninstall case.
  - **CLI**: existing `uninstall.ps1` flow, unchanged mechanism.
- Separately, a granular keep/delete checklist for local state, replacing the
  current single "remove config + database? y/N": venv (`.venv-nlm-v<version>`),
  `config.yaml`, the memory/index database (`repo-context.sqlite`), the local
  `.mcpb` copy, the `servers/` registry directory. **Must state explicitly that the
  memory/index database is one shared file across every repo you've ever indexed**
  (confirmed via `config.py:110-111` — single `~/.neo-localmcp/repo-context.sqlite`,
  not per-repo), not scoped to whatever repo the wizard happens to be run from.
- Keep the existing `Confirm-Destructive` typed-confirmation gate for any category
  that deletes data, applied per-category if the user picks a mix of keep/delete
  rather than one blanket confirmation covering silently-included categories.

### 9g — P5: retrieval-boost tuning + upgrade-persistence guarantee · **Opus, High**

Scope: `neo_localmcp/repo_memory.py` (`RETRIEVAL_BOOST_CAP`, `RETRIEVAL_BOOST_MIN_SHOWN`,
`retrieval_boost_retention_days` in `config.py`), `tests/test_upgrade_cycle.py`,
`tests/test_retrieval_memory.py`.

Explicitly gated on 9a-9f landing first, per the user: memory can't be trusted to
develop *or* to survive an upgrade until the surrounding install/uninstall surface
is solid enough that a user isn't fighting the installer while also trying to judge
whether memory helped.

- **Audit the wiring first, don't assume.** `record_task_query` (bumps `shown`) and
  `record_retrieval_feedback` (bumps `followed`/`corrected`) are already called
  live from `tools.py:849` and `tools.py:504` — confirm with a live multi-call
  session against a real repo that shown/followed/corrected counts actually move
  the way the design doc comment claims, not just in unit tests with mocked data.
- **Persistence-across-upgrade is likely already true by construction** —
  `repo-context.sqlite` lives directly under `APP_DIR`, never inside a
  `.venv-nlm-v<version>` directory, and `install.ps1`'s removal sweep only ever
  targets `.venv-nlm-v*`/`venvs`/`venv` paths. This has never been asserted by a
  test, though. Add one: seed `retrieval_boost` rows, run a real version-to-version
  upgrade via `install.ps1` (extending the existing `test_upgrade_cycle.py`
  pattern), assert the rows are byte-identical after. If this test fails, that's
  a real bug to fix before any tuning is meaningful — don't tune constants on top
  of an unverified persistence claim.
- **Then tune the constants** — `RETRIEVAL_BOOST_CAP = 8`, `RETRIEVAL_BOOST_MIN_SHOWN
  = 3`, `retrieval_boost_retention_days = 90` — against whatever the live-session
  audit above actually shows, not arbitrary adjustment. If the audit shows the
  signal is noisy at `MIN_SHOWN = 3`, raise it; if 90 days is clearly too long or
  short given real repo-visit cadence, adjust it. Document the reasoning for
  whatever changes, since these were originally picked without usage data either.
- Close out by updating `PROJECT_STATUS.md`'s "Can this be counted on to improve
  over time" framing — currently the honest answer is no (see 2026-07-01 session).
  After this phase it should be either yes with evidence, or a clearly stated
  reason it still isn't, not silence.

  picking this up tomorrow on their own.
- `OLLAMA_MODELS` env-var inheritance bug in `start_service()` — known, tracked in
  `PROJECT_STATUS.md`, not part of this plan.
- Section-summary cache not invalidated by code-version bump — known, tracked in
  `PROJECT_STATUS.md`, not part of this plan.
