# 1.0.7 Plan — enrichment reliability + graceful upgrade lifecycle

## Origin

A full-tilt cold-start test against AntiNotepad (Ollama killed, stray processes killed,
memory nuked, then exercised all 12 MCP tools live) surfaced two real, reproducible
problems on top of the working 1.0.5/1.0.6 deterministic-retrieval core:

1. **Ollama enrichment has no output bound.** `summarize_file` on a larger, code-heavy
   section (f4.7, 277 lines) ignored its own `keywords: at most 8` instruction and ran
   away into thousands of unrelated words until it hit the hard `eval_count: 4096`
   generation cap — 106 seconds for one call, and the garbage keywords get truncated to
   1000 chars and indexed into `repo_fts` as `kind='summary'`, meaning a later query for
   an unrelated term (e.g. "OAuth") could nonsensically surface this doc. Every
   Ollama-touching response also duplicates a 16-entry `installed_models` array across
   nested status blocks, wasting tokens on every call regardless of size.
2. **The upgrade path requires force-killing a live server.** Every `install.ps1` run
   this session hit a locked venv DLL because the previous side-by-side install's
   server process was still running, requiring a manual `Stop-Process -Force`. Doing
   this repeatedly across a session left visible DWM ghost window-frame artifacts on
   the host desktop (Task Manager screenshot, 2026-06-30) — non-graceful process
   teardown, not just an inconvenience. `install.ps1` also never prunes old timestamped
   venv directories; four accumulated from one session's testing alone.

## Phase breakdown

Phases run **in order, with an explicit pause between each** for a model/effort switch.
Do not start the next phase without the user resuming.

### 7a — Ollama enrichment bounding · **Sonnet, Medium**

Scope: `neo_localmcp/tools.py` (`_summarize_section`, `_split_summary_keywords`),
`neo_localmcp/ollama_client.py` (`chat`, request body construction).

- Add a `num_predict` cap (or Ollama's equivalent generation-length option) to the
  `/api/generate` request used by the section-summary prompt. Sized generously for
  "1-2 sentences + up to 8 keywords" (~300-400 tokens) — covers both the summary and
  keywords portions with one cap, not just keywords.
- Harden `_split_summary_keywords` to truncate keywords by **comma-count** (first ~8
  terms), not character count, so a still-under-cap-but-verbose response can't leak
  junk past the intended shape regardless of what the model outputs.
- If `eval_count` lands exactly on the `num_predict` ceiling, treat that as a runaway
  signal: mark the stored/returned result `truncated: true` rather than silently
  caching it as a considered answer. No auto-retry in this phase — just flag it;
  revisit auto-retry later only if flagging proves insufficient in practice.
- Trim the duplicated `installed_models` blocks out of nested `ollama_status` payloads
  in Ollama-touching tool responses — one copy is enough.
- New regression tests: a mocked "runaway" Ollama response (long, repeated garbage)
  proving the parser caps it at ~8 keyword terms regardless of input length; a
  contract test asserting `num_predict` is present in the outgoing request body; a
  test that `truncated: true` is set when `eval_count` hits the cap.

**Not in scope for 7a:** section chunking by length. The hypothesis is that a hard
generation cap alone neutralizes the failure mode independent of input size. Only add
chunking later if real usage shows the cap alone isn't enough.

### 7b — Graceful shutdown mechanism · **Opus, High**

Scope: `neo_localmcp/server.py`, `neo_localmcp/cli.py`, a new stop-coordination
primitive (likely alongside the existing `ollama-supervisor.json` state-file
convention in `neo_localmcp/ollama_client.py`).

This is the one genuinely open design question in 1.0.7 — the phase most likely to
need real judgment under ambiguity, which is why it's the one phase pulled up to Opus.

- Resolve how to interrupt FastMCP's blocking `mcp.run()` loop cleanly. Two candidate
  mechanisms to evaluate, not assume:
  - A portable **stop-request file** the server polls, consistent with the existing
    `ollama-supervisor.json` pattern — identical behavior on Windows/macOS/Linux,
    no OS-signal quirks.
  - Native **SIGTERM handling** via `signal.signal()` — simpler on macOS/Linux;
    historically inconsistent on Windows for processes not spawned with
    `CREATE_NEW_PROCESS_GROUP`. Verify before relying on it, don't assume it works.
- Land on one primary mechanism (or a primary + fallback) and implement graceful
  teardown: finish or cancel in-flight requests, close the SQLite connection cleanly,
  exit within a bounded window (~5-10s).
- Expose it as a new `neo-localmcp stop` CLI command that reads the tracked server PID
  from state and requests graceful shutdown, falling back to force-kill only after the
  bounded timeout, with a printed warning when that fallback path is actually taken.
- This benefits all three launch surfaces without separate work per-surface: the CLI
  stable launcher (used directly by Codex and by `install.ps1`/`install.sh`'s new
  upgrade flow in 7c), and indirectly the Claude Desktop `.mcpb` extension, since a
  cooperative server responds better to whatever termination Claude Desktop's own
  extension manager sends on upgrade, even though neo-localmcp doesn't control that
  outer trigger.

### 7b — DONE (2026-07-01). Handoff notes for 7c

Delivered: `neo_localmcp/lifecycle.py` (registry + per-PID stop-file watcher +
cross-platform liveness/force-terminate + `stop_servers` orchestration);
`server.main()` now registers on startup, runs a daemon watcher, and unregisters in
a `finally`; `cli.py` gained `neo-localmcp servers` (list) and `neo-localmcp stop`
(`--pid` / `--all` / `--match-executable` / `--timeout` / `--no-force`); `doctor`
now reports `running_servers`. 9 tests in `tests/test_lifecycle.py`, incl. a real
spawn-register-stop integration test. Verified live via the CLI: a spawned server
reports `stopped_gracefully` with no force-kill, ~instant.

Mechanism recap: the watcher calls `os._exit(0)` (a *self* exit -- releases file
locks, clean OS console teardown, no ghost frames) rather than any external kill.
`stop` is CLI-only by design (not an MCP tool) so a remote client can't kill servers.

**What 7c must do with this:**
- Upgrade flow should stop only the servers running from the venvs root being
  replaced, never Claude Desktop's separate `.venv`. Use:
  `neo-localmcp stop --match-executable "<AppHome>\venvs"` (matches on `sys.executable`).
  Prefer running that via a *already-installed* launcher; if none exists yet, invoke
  the source module directly.
- **Chicken-and-egg for the first 1.0.7 transition only:** servers currently running
  predate the watcher, so they won't be in the registry and `stop` can't reach them.
  Keep a bounded fallback in the installer for that one upgrade: attempt graceful
  `stop` first, then if the venv is still locked, fall back to the old detect-lock/
  force-kill path with a printed note. From 1.0.7 onward every future upgrade is
  graceful and this fallback should effectively never fire.
- Venv pruning: after a successful install, delete old `~/.neo-localmcp/venvs/<id>`
  directories, keeping the newest 1-2 for rollback. (Do NOT prune a venv whose server
  is still alive per the registry -- stop it first.)

### 7c — Upgrade flow rewrite (Windows) · **Sonnet, High**

Scope: `install.ps1`, `uninstall.ps1`, `neo_localmcp/cli.py` (wire the new `stop`
command in), regression coverage.

Depends on 7b's mechanism being proven first.

- Rewrite the upgrade sequence: graceful-stop old server (new `stop` command) → wait
  bounded timeout → build new venv → install → repoint the stable launcher → **prune
  old venv directories**, keeping the last 1-2 for rollback. (Four stale timestamped
  venvs accumulated from this session's testing alone — nothing currently garbage
  -collects them.)
- Force-kill remains available as an explicit last-resort fallback inside `stop`
  itself (from 7b), not as something the install script reaches for directly.
- New tests: spawn a real server subprocess, issue graceful stop, assert exit within
  budget with no orphaned children; run the full install flow end-to-end and assert
  exactly one live server process and zero stale venv directories afterward.

### 7c — DONE (2026-07-01). Handoff notes for 7d

Delivered: `install.ps1` now requests a graceful stop of servers under the venvs
root before building the new venv, and prunes old venv directories after a
successful install (`-KeepPreviousVenvs`, default 1). `uninstall.ps1` requests a
graceful stop before removing `venv`/`venvs`/`bin`, with a clear error (pointing at
manual Task Manager / Stop-Process) if that fails. Both scripts tolerate the
one-time chicken-and-egg case (a pre-1.0.7 server predates the stop watcher and
isn't reachable this way) by skipping/erroring with an explicit message rather than
crashing silently or corrupting state.

**Real finding during verification, fixed in both scripts:** even after a process
is *confirmed exited* (`pid_alive` false), Windows can hold its DLL file lock for a
brief moment longer -- deferred handle release / AV scan-on-close. A graceful stop
that genuinely succeeded could still see one transient `Remove-Item` failure
immediately afterward. Fixed with a short retry (5 attempts, 500ms apart) around
every removal in both scripts (`Remove-ItemWithRetry`). This is a Windows-specific
quirk worth knowing about for 7d: it may not exist on macOS/Linux at all (Unix
generally allows unlinking a still-open file), so don't port the retry blindly
without checking whether it's actually needed there.

Verified live, not just by test: killed real pre-1.0.7 stray servers, confirmed the
one-time fallback path skips them with a clear message instead of aborting; then a
clean install -> spawn a genuine live 1.0.7 server -> `uninstall.ps1` while it was
running succeeded (exit 0, no lock error, no force-kill) and the process was
confirmed fully gone afterward. `tests/test_upgrade_cycle.py` (new, Windows-only,
skipped elsewhere) automates exactly this: real `install.ps1` against an isolated
`NEO_LOCALMCP_HOME`, spawns a real server, runs `uninstall.ps1` while it's live,
asserts clean exit and zero leftover `venvs`/`bin`.

**What 7d must do with this:**
- Mirror the graceful-stop-before-remove/build structure in `install.sh`/`uninstall.sh`,
  using `neo-localmcp stop --match-executable <venvs path>` the same way.
- Do NOT assume the DLL-delayed-release retry is needed -- test whether a bare
  `rm -rf` on a just-exited process's venv ever actually fails on macOS/Linux before
  porting the retry loop. If it doesn't reproduce, don't add complexity for nothing.
- The rest of the mechanism (registry, stop-file, `os._exit(0)` self-exit) from 7b
  is already fully cross-platform -- nothing there needs porting, only the shell
  script orchestration around it.

### 7d — macOS symmetric changes · **Sonnet, High** · DEFERRED by user request (2026-07-01)

Scope: `install.sh`, `uninstall.sh`. Not started. The Windows side (7a-7c) is
released as 1.0.7; `install.sh`/`uninstall.sh` still have the pre-1.0.7
detect-lock/no-graceful-stop behavior. This is a known gap, not an oversight --
revisit when there's a reason to (mac usage, or just picking the plan back up).

When resumed:
- Mirror 7c's flow in the shell scripts.
- Flag explicitly (not silently assume): the Windows-specific file-lock-on-delete
  problem that motivated all of this may not reproduce the same way on macOS/Linux,
  since Unix generally permits unlinking a file that's still open. The graceful-stop
  discipline is still good practice cross-platform, but the *urgency* that drove 7b/7c
  is Windows-specific.
- Do NOT port the 7c DLL-delayed-release retry loop assuming it's needed -- test
  whether a bare `rm -rf` on a just-exited process's venv ever actually fails on
  macOS/Linux first.
- Cannot be runtime-verified without macOS hardware. Ship the symmetric script changes,
  but call out plainly that they need a human with a Mac to smoke-test before this
  phase is considered actually done, not just "written."
- The lifecycle mechanism itself (`neo_localmcp/lifecycle.py`: registry, stop-file,
  `os._exit(0)` self-exit) from 7b is already fully cross-platform and needs no
  changes -- only the shell script orchestration around it is missing.

## Deferred (explicitly, not forgotten)

- **7d** (see above) -- macOS install/uninstall script parity.
- **P5 tuning** (the implicit-feedback retrieval boost from 1.0.6): architecturally
  correct and unit-tested, but hasn't accumulated real usage against a live repo yet.
  Needs mileage before it's worth revisiting, not more design work right now.
- Length-based section chunking for summarization (only if 7a's generation cap proves
  insufficient in practice).
- VRAM-thrash awareness (alternating "ranking" vs "summary" purpose Ollama calls costs
  a real ~10-20s reload each time on hardware that can't keep both models resident).
  Noted, not a bug, not in scope here.

## Process note for whoever runs 7b-7d

Testing the upgrade flow will require spawning and stopping real server processes
repeatedly. Prefer the new graceful `stop` command over `Stop-Process -Force` once 7b
lands — that's the entire point of this plan. If a force-kill is genuinely needed
during development/debugging, treat it as the last resort it's meant to be, not the
default testing loop.
