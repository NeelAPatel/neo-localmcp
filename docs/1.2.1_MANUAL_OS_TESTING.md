# v1.2.1 Manual macOS and Windows Test Guide

## Status and safety rules

This guide is the manual acceptance handoff for the v1.2.1 issue branches.
Automated tests do not replace these checks. Mark a checkpoint complete only
after it has been run on the named operating system.

- Do not merge or push anything to `main` from this test session.
- Issue branches merge into `dev/v1.2.1_implementation` after their automated
  checks are green; manual OS verification may remain pending and must be
  recorded as such in the PR/integration notes.
- Preserve `~/.neo-localmcp` memory, SQLite, config, clients, logs, and cache
  unless a test explicitly uses `--delete-memory` in an isolated test home.
- Run real lifecycle checks from a fresh terminal window. Do not treat a
  shell that inherited a temporary PATH modification as PATH evidence.

## Preparation on each machine

1. Check out the issue branch or `dev/v1.2.1_implementation` being tested.
2. Record any existing installation and registrations:

   ```sh
   python3 setup.py install --dry-run
   neo-localmcp config clients status
   ```

3. For destructive experiments, set `NEO_LOCALMCP_HOME` to a disposable path;
   do not use the real managed root.
4. Start a new terminal after a PATH update. On Windows, open a new Command
   Prompt or PowerShell after the per-user `Path` change.

## 12.1a — #100 PATH hint and opt-in registration

### macOS

1. Run a normal install or reinstall with no PATH flag:

   ```sh
   python3 setup.py install
   # or: python3 setup.py reinstall
   ```

2. Confirm the success output prints the exact `export PATH=".../venv/bin:$PATH"`
   command and does not modify the shell rc file.
3. Run the same operation with `--add-to-path`:

   ```sh
   python3 setup.py reinstall --add-to-path
   ```

4. Confirm the detected shell file (`~/.zshrc`, `~/.bashrc`, or Fish
   `~/.config/fish/config.fish`) contains exactly one marked
   `# neo-localmcp PATH` block.
5. Open a new terminal and verify:

   ```sh
   command -v neo-localmcp
   neo-localmcp doctor
   ```

6. Confirm the resolved executable is the managed
   `~/.neo-localmcp/venv/bin/neo-localmcp`, not a stale system installation.

### Windows

1. Run `python setup.py install` or `python setup.py reinstall`; confirm the
   success output prints the `setx PATH "%PATH%;...\\venv\\Scripts"` hint.
2. Run `python setup.py reinstall --add-to-path`.
3. Open a new Command Prompt or PowerShell and verify:

   ```powershell
   Get-Command neo-localmcp
   neo-localmcp doctor
   ```

4. Confirm the per-user PATH contains the managed `venv\\Scripts` directory
   once and the command does not resolve to a stale installation.

## 12.1b — #103 client-selection parity

Run these checks on macOS and Windows after the #103 branch is merged into
the integration branch.

1. Start with a deliberately non-fresh but safe installation state and choose
   an explicit Install client set. Confirm that exact set is registered; it
   must not be replaced by a disk snapshot.
2. Run Reinstall and change one client directly (for example, add Codex to a
   Claude-Code-only setup). Confirm the selected registrations change without
   using the separate Manage connected clients operation.
3. Run partial client detach, for example:

   ```sh
   python3 setup.py uninstall --client codex
   ```

   Confirm only Codex is detached; the runtime and durable memory remain so
   other client registrations never point at a deleted launcher.
4. Run plain `uninstall` with no `--client`. Confirm every registration is
   detached, the runtime is removed, and durable memory remains.
5. Confirm memory deletion still requires the separate explicit
   `--delete-memory` full-wipe path and its confirmation.

### Wizard parity (`python3 setup.py` interactive / `setup_wizard.py`)

6. Launch the interactive wizard, choose **Reinstall / update**. Confirm a
   client-selection screen now appears (previously reinstall skipped it and
   silently kept the on-disk set), pre-filled with the currently-registered
   clients. Deselect one client and confirm only that change is applied.
7. Launch the wizard and choose **Uninstall**. Confirm a new first screen
   offers "Uninstall" vs. "Detach specific clients only". Choosing "Detach
   specific clients only" must show the same client multi-select, skip the
   runtime-only/full-wipe screens entirely, and leave the runtime and all
   data untouched after only the chosen clients are removed.
8. Choosing plain "Uninstall" from that new screen must behave exactly as
   before: runtime-only vs. full-wipe question, then removal of every
   registered client.

## 12.1c — #68 Ollama environment inheritance

### Windows local Ollama

1. Configure `OLLAMA_MODELS` to the external models drive before launching
   the Claude Desktop `.mcpb` extension.
2. From that extension-host launch path, run the neo-localmcp doctor/model
   status flow.
3. Confirm the configured models are listed from the external drive; the
   result must not fall back to an empty default Ollama models directory.
4. Confirm a missing model is still never auto-downloaded.

### macOS remote Ollama

1. Use the existing remote/Tailscale Ollama base URL.
2. Confirm the application continues to refuse starting a remote Ollama
   service locally and that normal status/context operations remain unchanged.

## 12.1d–f — #101 recommendations and pull warning

Run on both macOS and Windows after the #101 branch is merged into the
integration branch.

1. Open each fast, summary, and embedding model-picker page.
2. Confirm **RECOMMENDED FOR YOUR SETUP** appears before **ALL INSTALLED
   MODELS**, the installed list remains complete, and numbering spans both
   sections continuously.
3. Confirm the static recommendations are exactly:

   | Role | First | Second |
   |---|---|---|
   | Fast | `qwen3.5:4b` | `qwen3.5:9b` |
   | Summary | `gemma4:12b` | `qwen3:8b` |
   | Embedding | `bge-m3:latest` | `mxbai-embed-large:latest` |

4. Verify installed recommendations show `[x] available`; unpulled ones show
   `[ ] not pulled`.
5. Select at least one unpulled recommendation and reach the confirmation
   page. Confirm it lists the exact `ollama pull <model>` command but does not
   block proceeding.
6. Select only installed models and confirm the pull-warning block is absent.

## 12.1h — #114/#112/#113 client-registration reliability

**Status: executed 2026-07-21, agent-run, real end-to-end (not `--preview`),
both environments below — PASSED.** A second bug beyond #114 was found and
fixed in the process (`_install_like`'s client-intent pre-write was
defeating `apply_client_selection`'s diff; see `docs/1.2.1_PLAN.md`'s 12.1h
entry). The steps below are kept as-written for anyone who wants to
reproduce by hand; only the real Claude Desktop `.mcpb` GUI install (Settings
> Extensions) was left undone, since that step can't be driven headlessly.

Two environments, both on macOS, differing only in where Ollama runs — the
bug and fix are entirely in client reconciliation, not Ollama, so the two
runs exist to get real hands-on coverage twice, not because Ollama's
location matters to this fix:

- **Environment A — Ollama on localhost** (MacBook Pro).
- **Environment B — Ollama remote on NeoRyzen** (point `base_url` at
  NeoRyzen the same way you already do for other remote-Ollama setups).

Skip re-running 12.1a (PATH), 12.1c (env inheritance), and 12.1d-f
(recommendations) here — those are already verified and this change doesn't
touch any of that code. 12.1b's non-wizard CLI checks (`--client` flags on
reinstall/uninstall) are also unaffected and don't need a re-run; only the
**wizard-driven install** path changed.

### Step 0 — find out what's already on each machine

Before touching anything:

```sh
neo-localmcp where
neo-localmcp config clients status
cat ~/.neo-localmcp/clients/registrations.json 2>/dev/null
```

This tells you whether `~/.neo-localmcp` already exists on that machine (it
almost certainly does, from earlier 1.2.1 testing) — that matters, because
the bug this phase fixes **only reproduces when a managed root already
exists but no client is registered yet**, not on a genuinely first-ever
install. A truly empty machine would have masked this bug the whole time.

### Step 1 — reproduce the exact bug conditions, then confirm the fix

Do this on an isolated test home first so you're not gambling with real
`~/.neo-localmcp` state:

```sh
export NEO_LOCALMCP_HOME=/tmp/nlm-1218-test   # or an equivalent disposable path
rm -rf "$NEO_LOCALMCP_HOME"
mkdir -p "$NEO_LOCALMCP_HOME"
python3 setup.py install --dry-run    # confirm it reports ABSENT
python3 setup_wizard.py               # top-level entrypoint, see README.md
```

1. Run the wizard through **Install**, select Claude Code (and Codex if you
   have the real CLI installed) as clients, let it complete.
2. Immediately after, without wiping `NEO_LOCALMCP_HOME`, run the wizard
   again and choose **Reinstall**, this time changing the client selection
   (e.g. add/drop Codex).
3. After each run, confirm:
   ```sh
   cat "$NEO_LOCALMCP_HOME/clients/registrations.json"
   ls "$NEO_LOCALMCP_HOME"  # sanity: managed root exists
   ```
   the registrations file lists exactly the clients you selected — never
   empty when you selected at least one client, and never silently reverted
   to whatever was there before.
4. For Claude Code specifically, confirm the slash-command files actually
   landed:
   ```sh
   ls ~/.claude/commands/neo-localmcp/   # real home, not NEO_LOCALMCP_HOME
   claude mcp list                       # should show neo-localmcp
   ```
   (Note: slash commands and `claude mcp list` read from your **real** home
   and real `claude` CLI regardless of `NEO_LOCALMCP_HOME` — that's expected;
   this is why the isolated-home run above only proves the registrations
   *record* is correct, and this step proves the actual client-facing effect.)

Unset `NEO_LOCALMCP_HOME` when done with the isolated run.

### Step 2 — same check against real state

Once the isolated run passes, do one real (non-isolated) wizard
**Reinstall** on your actual `~/.neo-localmcp` install, changing at least one
client selection, and confirm the same way: `claude mcp list` shows
`neo-localmcp` if Claude Code was selected, `~/.codex/config.toml` (or your
`CODEX_HOME` path, see below) has a `[mcp_servers.neo-localmcp]` block if
Codex was selected.

### Step 3 — #113 (Codex `CODEX_HOME`)

Only meaningful if you have a real Codex CLI/Desktop install to cross-check
against:

1. Check whether `CODEX_HOME` is set in your normal shell environment
   (`echo $CODEX_HOME`). Note whatever you find — this determines which case
   below is your "real" case.
2. If unset: confirm `neo-localmcp config clients setup --client codex`
   writes to `~/.codex/config.toml` (unchanged fallback behavior) and that
   your real Codex CLI reads it correctly (`codex` or whatever check you
   normally use to confirm MCP servers are registered).
3. If you want to exercise the `CODEX_HOME`-set branch specifically without
   touching your real Codex config, run once with it overridden:
   ```sh
   CODEX_HOME=/tmp/nlm-codex-home neo-localmcp config clients setup --client codex
   cat /tmp/nlm-codex-home/config.toml   # should contain the neo-localmcp block
   ```
   This doesn't need a real Codex CLI to validate — it just proves the
   installer honors the env var instead of hardcoding `~/.codex`.

### Report back

For each environment (A: localhost Ollama, B: NeoRyzen Ollama), record:
pass/fail on Steps 1-3, the exact `registrations.json` contents you saw, and
whether `claude mcp list` / the Codex config block appeared where expected.
If either environment fails, capture the full scrolling wizard output (not
just the final screen) before filing a follow-up — that visibility gap is
tracked separately as #115, deliberately not fixed yet, so today it's the
only way to see what happened mid-run.
