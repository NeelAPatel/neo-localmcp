# neo-localmcp 1.0.10 Cross-Platform Setup v2 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Replace the Windows-first installer family with one Python 3.12+ lifecycle that safely installs, reinstalls, uninstalls, clean-installs, migrates, reconnects clients, preserves memory by default, and verifies the resulting MCP endpoint on Windows, macOS, and Linux.

**Architecture:** `setup_v2.py` is the only lifecycle entrypoint during development and delegates to focused modules under `neo_localmcp/installer/`. Shared Python code owns policy; cross-platform libraries and small platform adapters handle executable paths, process details, permissions, and filesystem replacement. A candidate venv is built and verified before any healthy runtime is replaced.

**Tech Stack:** Python 3.12+, stdlib `argparse`/`venv`/`pathlib`/`json`/`subprocess`, `psutil` for cross-platform process ownership and trees, existing MCP/FastMCP client stack, pytest, isolated temporary homes, and real lifecycle smoke tests on macOS/Windows/Linux.

## Global Constraints

- `setup_v2.py` is the only supported install/reinstall/uninstall entrypoint until it is renamed to `setup.py` after cross-platform parity.
- The installed executable remains `neo-localmcp`; no `neo` alias is created.
- Python 3.12 or newer is required; the installer never installs system Python.
- Installation source is the local checkout containing `setup_v2.py`.
- `install` reuses durable data; `reinstall` replaces only the runtime; `uninstall` removes only the runtime by default.
- `install --clean` deletes the entire validated managed root and then installs fresh.
- `uninstall --delete-memory --yes` deletes the entire validated managed root and does not reinstall.
- Full-root deletion always requires interactive confirmation or the explicit `--yes` non-interactive flag.
- Durable data is never deleted by default.
- Neo unloads models it used but never kills a shared Ollama daemon.
- Process termination requires verified Neo ownership; never kill by process name alone.
- Avoid OS-specific files unless they materially improve reliability, stability, or functionality.
- When implementation depends on a library/tool API, consult Context7 first when available, then the product's primary official documentation; use third-party sources only when primary documentation is silent and label the inference.
- Every command and subcommand supports `-h` and `--help`.
- Implementation is test-first, each phase ends in a focused commit, and old installers remain available until real Windows parity is verified.

## Scope and sequencing

This plan is intentionally sequential because paths, state, runtime transactions,
and ownership are safety boundaries consumed by every later phase. Tasks 1-13 can
be built and verified on macOS now. Task 14 uses Windows x64 CI as its durable gate
and any convenient temporary Windows VM only for targeted live smoke testing. Task
15 adds Linux automation. The old scripts are retired only in Task 16 after all
gates pass.

## Documentation verification protocol

Before implementing an unfamiliar or version-sensitive API:

1. Resolve and query the official/high-reputation Context7 library entry.
2. Confirm safety-sensitive behavior against the upstream project's official docs
   or repository documentation when Context7 excerpts leave ambiguity.
3. Record the relevant documentation URL in task notes or a code comment only
   where it explains a non-obvious safety decision.
4. If primary documentation does not answer the question, state the inference and
   add a test that makes the assumed behavior explicit.

Initial documentation checks completed while writing this plan:

- psutil officially documents PID/create-time identity, `exe()`, `cmdline()`,
  descendant traversal, `terminate()`, `kill()`, `wait_procs()`, and handling of
  `NoSuchProcess`/`AccessDenied`: https://psutil.readthedocs.io/
- Ollama officially documents model unload through `POST /api/generate` with the
  model name and `keep_alive: 0`, without stopping the daemon:
  https://github.com/ollama/ollama/blob/main/docs/api.md

### Task 12: Orchestrate install, reinstall, uninstall, and clean install

**Purpose:** Compose approved primitives into the exact lifecycle semantics.

**Model / effort:** GPT-5.5 High; this is the highest destructive-policy phase.

**Files:**
- Create: `neo_localmcp/installer/operations.py`
- Create: `tests/installer/test_operations.py`
- Modify: `neo_localmcp/installer/__init__.py`

**Interfaces:**
- Produces: `install(context, clean=False, assume_yes=False) -> OperationResult`.
- Produces: `reinstall(context) -> OperationResult`.
- Produces: `uninstall(context, delete_memory=False, assume_yes=False) -> OperationResult`.
- `OperationContext` contains paths, source root, Python executable, reporter, process provider, clock, and confirmation function.

- [ ] **Step 1: Write ordered-call tests with fake components**

Assert install order: validate source/Python → detect state → stop owned processes if needed → unload models → snapshot clients → migrate → build/validate candidate → promote → restore clients → verify endpoint → complete metadata → summary.

- [ ] **Step 2: Write semantic matrix tests**

Cover all state/operation combinations. Required assertions:

| Operation | Removes venv | Recreates venv | Preserves durable data | Active clients afterward |
|---|---:|---:|---:|---|
| `install` | only when replacement needed | yes | yes | restored/selected |
| `reinstall` | yes | yes | yes | restored |
| `uninstall` | yes | no | yes | removed |
| `install --clean` | yes | yes | no | newly selected only |
| `uninstall --delete-memory` | yes | no | no | removed |

- [ ] **Step 3: Write confirmation and cancellation tests**

`--clean` and `--delete-memory` refuse non-interactive deletion without `--yes`; interactive cancellation performs no destructive call and records `cancelled`.

- [ ] **Step 4: Implement default install/reinstall/uninstall**

Default uninstall removes only `venv/` after client cleanup. It does not describe or execute venv recreation. Reinstall never deletes durable directories.

- [ ] **Step 5: Implement full wipe and clean install**

Validate the root immediately before deletion, not only at operation start. Clean install recreates the root/layout after deletion and does not reuse deleted client records.

- [ ] **Step 6: Implement recovery paths**

If client restoration fails after successful runtime promotion, keep the runtime, fail the operation visibly, and print exact recovery instructions. If promotion fails, restore the previous runtime and prior registrations.

- [ ] **Step 7: Run operation and full suites**

Run: `python -m pytest tests/installer/test_operations.py -q && python -m pytest -q`

- [ ] **Step 8: Commit**

```bash
git add neo_localmcp/installer/operations.py neo_localmcp/installer/__init__.py tests/installer/test_operations.py
git commit -m "feat: orchestrate safe setup v2 lifecycle operations"
```

**Gate:** The semantic matrix is executable and default uninstall cannot reach any durable-root deletion function.
