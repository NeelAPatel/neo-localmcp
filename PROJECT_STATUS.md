# Project Status

Updated: 2026-07-02

## Current phase

1.0.10 setup lifecycle work (Phases 1-16) is complete. `setup.py` is the sole supported lifecycle entrypoint on macOS and Windows; Linux is explicitly deferred. Legacy installer scripts are archived under `_LegacyInstallers/`. Native Python 3.12 CI is green on macOS and Windows (run 28632598884), and the real managed installation is running 1.0.10 from a clean canonical layout.

`setup.py` installs, reinstalls, default-uninstalls while preserving durable data, clean-installs, and full-wipes from a local checkout. It builds and verifies a candidate runtime before promotion, stops Neo-owned processes, unloads only Neo-used Ollama models, migrates recognized legacy layouts, reconnects explicitly selected/recorded clients, and verifies the installed CLI/MCP endpoint.

1.0.9 onboarding-polish work (`docs/1.0.9_PLAN.md`) is complete and released. Version 1.0.10 is synchronized across `neo_localmcp/__init__.py`, `pyproject.toml`, and the Claude Desktop manifest; its bundle is rebuilt during this closeout.

## Acceptance targets

- At least 50% fewer discovery/read tokens on representative coding tasks.
- At least 30% fewer total task tokens without lower edit or test accuracy.
- Complete indexes explicitly report completeness and prune vanished files safely.
- Ollama failures never suppress deterministic repository context.
- Normal Ollama operations have bounded health, start, warm, and inference deadlines.

## Known limitations

- Symbol extraction remains regex-based rather than a full Tree-sitter structural index.
- Token counts are estimated from returned characters until client usage telemetry is available.
- Compatibility alias `context_prepare` remains exposed for one release.
- Linux setup lifecycle and CI support are deferred beyond 1.0.10.
- Section-summary caching is keyed on source-file content hash only, not on the neo-localmcp code version -- a cache entry generated by a buggy older version is not automatically invalidated when the bug is fixed; only a source-file content change (or manual cache clear) will regenerate it.
- `start_service()` may silently use the wrong Ollama models directory depending on process ancestry (see Remaining validation above).
- **Fixed 2026-07-03**: `utils.rg_search`'s pure-Python fallback (used whenever `rg` is not on PATH -- e.g. GitHub's `macos-latest` CI runner has no ripgrep preinstalled, unlike `windows-latest`) did a literal substring match of the *entire* regex-syntax query string tools.py always sends it (`"(?:term1|term2)"`), which never occurs verbatim in real source text -- so batched search silently returned zero results on any host without `rg`, for single- or multi-term queries alike. Surfaced by CI (not local dev, since `rg` was present there) after the #2 fix above added a test relying purely on batched-search matching. Now compiles and searches the query as an actual regex. Covered by `tests/test_utils.py::test_rg_search_fallback_interprets_the_query_as_a_regex`.
- Retrieval-boost memory: "can this be counted on to improve retrieval over time?" is now a qualified yes, not the previous no. It provably works and survives upgrades (see Verified), and it can never hurt correctness -- it's capped far below structural evidence and never penalizes silence. But its impact is modest by design: it only nudges once the *same task string* recurs >= 3 times in one repo, so it helps repeated workflows in a repo, not first-time or one-off tasks, and its real-world benefit has still only been exercised in one controlled audit session, not measured across sustained multi-session usage. It is a safe, bounded nudge, not a broadly-learning ranker.
- **Fixed 2026-07-03**: `config.py`'s default `repo.exclude_dirs` (`.venv`, `venv`, etc.) did exact directory-name matching only, not prefix/glob matching, so a differently-named local venv or vendor directory (e.g. a disposable `.venv-phase14` test venv, or this project's own installer convention `.venv-nlm-v<version>`) got indexed and its vendored dependency code could outrank real repository source. `utils.scan_repo_files` now matches `exclude_dirs` entries as `fnmatch` glob patterns (plain names still match exactly); defaults broadened to `.venv*`/`venv*`. Covered by `tests/test_utils.py`.
- **Fixed 2026-07-03**: when a candidate file had multiple independent hint locations for different matched terms, `tools.py`'s excerpt-centering logic took whichever hint sorted first by line number rather than the hint tied to the term that actually matched the query (reproduced live: a query for `num_predict` matched `ollama_client.py` at line 308, but the shown excerpt centered on an unrelated earlier line 28). `_add_candidate` now tracks a `line_hint_weights` map (line number -> strongest evidence weight) and centering picks the highest-weight hint (ties break to the earliest line for determinism), falling back to the old first-hint behavior only when no weighted hint exists. Retrieval-boost's "followed" scoring inherited the same bug (`repo_memory.record_retrieval_feedback` only checked overlap against the one chosen window); fallback-centered `sections_for_memory` entries now also carry the file's other listed hint lines (`hint_lines`), and a `file_excerpts` pull overlapping any of them counts as followed, not just the primary window. Covered by `tests/test_context.py::test_excerpt_centers_on_highest_weight_hint_not_first_by_line` and `tests/test_retrieval_memory.py::test_follow_up_pull_on_a_secondary_hint_still_counts_as_followed`.
- **Fixed 2026-07-03**: a genuinely fresh clone could not follow the README's own `python3 setup.py install` instructions on a bare system Python. `setup_cli.py` unconditionally imports `neo_localmcp.installer`, which imports `installer/processes.py`'s top-level `import psutil` -- so the *host* interpreter invoking `setup.py` (not the managed runtime it builds) needed `psutil` importable before anything else ran, with no bootstrap step and no README mention. This had been masked because CI's `pip install -e ".[test]"` step always installs the full dependency set (including `psutil`) before any lifecycle test runs. Reproduced live with a genuinely bare venv (`ModuleNotFoundError: No module named 'psutil'` from `setup.py install --dry-run`) and confirmed the fix resolves it (same bare venv, `psutil` installed first, dry-run completes cleanly). Fixed by documenting the prerequisite in `README.md`'s Install section (`python3 -m pip install "psutil>=5.9,<8"` before running `setup.py`) rather than changing import structure, since `psutil` is needed almost immediately regardless.
