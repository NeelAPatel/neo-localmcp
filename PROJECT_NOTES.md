I end up restarting my PC"). Found and fixed a real bug during live interactive testing: the "register with Claude Code/Codex?" prompt was skipped by an early `return` on the same-version-already-installed path, causing a leftover piped answer to be misinterpreted as a menu choice on the next loop iteration. Verified live: status detection, the same-version skip path, and the typed-confirmation safety gate (a wrong phrase correctly cancels, leaving the venv/config/database untouched — confirmed by direct filesystem check afterward, catching my own `ls` mistake of forgetting `-a` for the dotfile venv directory along the way).

## 2026-07-01 (2)

- Released 1.0.8: live post-restart testing of 1.0.7 through the connected Claude Desktop extension surfaced two more real findings, plus a design change from the user in response. (1) A stale pre-1.0.7 cached section-summary (1000-char keyword truncation, timestamped before the P7a fix) was served as `cached: true` — expected behavior since cache invalidation is keyed on source-file content hash, not on a code version, but worth knowing operationally: 1.0.7's bounding fix does not retroactively clean already-cached garbage. (2) Ollama auto-start silently used the wrong models directory after a fresh PC restart: the user's `OLLAMA_MODELS` env var (pointing at a large external drive) was not inherited by the server process spawned under Claude Desktop's process tree, so it fell back to a near-empty default path and reported `model_missing` with no error — a real bug in `ollama_client.py`'s `start_service()`, not yet fixed, tracked as a 1.0.9+ follow-up.
- The user then asked why this session's repeated `install.ps1` runs spiked C: drive utilization despite Ollama's models living on a different drive entirely. Root cause: 1.0.7's P7c side-by-side venv scheme still created a brand-new timestamped `venvs/<timestamp>-<pid>` directory on every install run (8-10+ times across the session) and did a full `pip install --force-reinstall` of ~35 dependencies into each one — thousands of small file writes per build, all on `C:\Users\Neel\.neo-localmcp`, compounded by Windows Defender scanning each newly-written file. This diagnosis led directly to 1.0.8's redesign.
- **Single-venv-per-version redesign** (Windows only; macOS `install.sh`/`uninstall.sh` still on the old scheme, same 7d gap as before): now that 1.0.7's P7b/P7c graceful-stop mechanism can reliably stop a running server before touching its files, the original justification for "always create a new directory, never touch an existing one" (dodging a locked-file failure against a live server) no longer applies. `install.ps1` now maintains exactly one venv at `~/.neo-localmcp/.venv-nlm-v<version>`: reinstalling the same version is a fast no-op (~1.2s, verified live, vs. 40-90s for a full rebuild) unless `-Repair` is passed; installing a different version gracefully stops any running server, removes the old venv (including sweeping the entire legacy pre-1.0.8 `venvs/` side-by-side layout wholesale on first upgrade), and builds fresh. `uninstall.ps1` updated to match. `-KeepPreviousVenvs`/`-Upgrade` removed (no longer meaningful — there is never more than one venv to retain). Verified live end-to-end against the real `~/.neo-localmcp`: a real upgrade from a leftover 1.0.7 `venvs/` folder (2 stale dirs, ~158MB) correctly swept the whole legacy folder and landed on a single `.venv-nlm-v1.0.8`. New tests: `test_reinstalling_same_version_is_a_fast_noop_not_a_rebuild` (proves no rebuild via directory ctime / python.exe unchanged), `test_repair_flag_forces_rebuild_of_same_version` (proves `-Repair` actually rebuilds, using file ctime since Windows' `venv` module copies `python.exe` preserving the source interpreter's mtime, not build time — a real test-design trap worth remembering).

## 2026-07-01

- Released 1.0.7 after a full-tilt live cold-start test against AntiNotepad (Ollama killed, memory nuked, all 12 MCP tools exercised end-to-end) surfaced two real problems the automated suite hadn't caught: an unbounded Ollama section-summary generation (106s runaway on a code-heavy section, ignoring its own "at most 8 keywords" instruction, polluting the FTS index with garbage) and an upgrade path that required force-killing a live server to unlock its venv, which repeatedly left DWM ghost window-frame artifacts on the host desktop.
- P7a: bounded section-summary generation with `num_predict` (config: `ollama.section_summary_num_predict`, default 400), covering both the summary and keywords portions with one cap. A response that hits the cap is flagged `truncated: true` and skipped from caching/FTS-indexing entirely rather than silently poisoning the index. Added a parsing-layer defense independent of the cap: keywords are capped by comma-count, not character count. Removed duplicate `installed_models` blobs from every Ollama-touching response (summarize whole-file, summarize section, ranking-assisted context).
- P7b: added `neo_localmcp/lifecycle.py` — a cross-process graceful-stop mechanism. Each server self-registers on startup and runs a daemon thread polling a per-PID stop-file; on trigger it does best-effort cleanup and calls `os._exit(0)`, a process-initiated exit that releases file locks and tears down cleanly, unlike an external terminate (on Windows, `SIGTERM` via `os.kill` *is* `TerminateProcess` — the exact mechanism that caused the ghost frames — and FastMCP installs no signal handlers to intercept it). New CLI commands `neo-localmcp servers` (list) and `neo-localmcp stop` (`--pid`/`--all`/`--match-executable`/`--timeout`/`--no-force`), CLI-only by design so a remote MCP client cannot stop servers. `doctor` now reports `running_servers`.
- P7c: rewired `install.ps1`/`uninstall.ps1` to request a graceful stop before touching venv files, with the old force-kill-on-lock-error behavior kept only as an explicit, clearly-messaged last resort for the one-time case of a pre-1.0.7 server that predates the stop watcher. `install.ps1` now prunes old side-by-side venv directories after a successful install (`-KeepPreviousVenvs`, default 1) instead of accumulating them forever. Found and fixed a real Windows quirk during live verification: even after a process is confirmed exited, Windows can hold its DLL file lock for a moment longer (deferred handle release / AV scan-on-close), so both scripts now retry a removal briefly before treating it as a genuine lock. Verified live: a real 1.0.7 server was spawned, `uninstall.ps1` was run while it was still running, and it stopped gracefully with no lock error and no force-kill — the exact failure mode that motivated this work, now closed. Automated as `tests/test_upgrade_cycle.py` (Windows-only, real install/spawn/uninstall subprocess integration test).
- 7d (macOS `install.sh`/`uninstall.sh` parity) explicitly deferred by user decision — not started, not forgotten; tracked in `docs/1.0.7_PLAN.md`.
- P5 (the 1.0.6 implicit-feedback retrieval boost) remains deferred pending real usage mileage, unrelated to this release's scope.
